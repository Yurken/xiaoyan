# Development Principles

本项目的默认开发目标是：低耦合、高内聚、可渐进重构。新增功能不能再通过继续堆大文件来完成。

## 结构约束

- 页面文件只负责组装视图、路由级状态和区块组合，不承载成组的远程调用、副作用和数据整形。
- 任何功能一旦同时包含 `状态 + 异步请求 + JSX 区块`，必须拆到 `features/<domain>/` 下。
- 同一功能域内的常量、纯函数、类型和映射统一收敛到 `shared.ts`，禁止在多个页面重复拷贝。
- 弹窗、工作区、面板、结果卡片优先做成独立组件；涉及文件系统、Tauri、网络、节流、防抖的逻辑优先做成 `use*.ts` hook。

## 文件规模约束

行数限制是参考值而非死规则，核心目标是**避免把不相关的职责堆在同一文件**，而不是机械地卡行数。

- 新增页面目标不超过 `500` 行，超过 `800` 行时必须说明理由并拆分出至少一个 feature 组件或 hook。
- 新增 feature 组件目标不超过 `400` 行，超过 `600` 行时应将独立子逻辑拆成更小的组件或 hook。
- 新增 hook 目标不超过 `300` 行；超过后优先考虑能否拆成更小的状态机、适配器或 helper。
- 现有遗留大文件如果已经远超上限，后续改动**不得继续向文件内部追加整块功能**；应先抽离本次触达的逻辑，再继续开发。

## 耦合控制

- 页面不直接编排多个不相关子域的状态。每个子域应有自己的 hook 或 controller。
- UI 组件通过 props 接收数据和动作，不直接依赖 `apiClient`、`journalApi`、Tauri 文件接口。
- 一个模块只对一个理由变化：视图变化进组件，交互状态进 hook，跨模块复用逻辑进 `shared.ts` 或 `lib/`.
- 禁止在多个页面维护同一份字段映射、状态机或请求参数构造逻辑。

## 修改策略

- 修改大文件时，优先执行“抽离后修改”，不要执行“先把新逻辑塞进去，之后再整理”。
- 新增能力时，优先复用已有 feature 目录；没有合适归属时，先创建功能目录，再落代码。
- 引入新状态前，先判断它是否属于已有 hook；如果属于，扩展 hook，而不是扩展页面。
- 新增弹窗前，先判断它是否可以作为独立组件存在；默认答案应是可以。

## 命名约定

- **文件名**：页面用 `PascalCase.tsx`；hook 用 `use<Domain><Purpose>.ts`；共享逻辑用 `shared.ts`；常量/纯函数用 `utils.ts` 或 `helpers.ts`。
- **组件后缀**：弹窗以 `Modal` 结尾，工作区以 `Workspace` 结尾，面板以 `Panel` 结尾，设置分区以 `Section` 结尾。命名要能从文件名推断出职责，禁止使用 `Component`、`Widget`、`Item` 等无意义后缀。
- **Hook 命名**：`use<名词>` 表示数据/状态管理，`use<动词>` 表示操作封装（如 `useSubmissionForm` vs `useExportPdf`）；同一功能域的多个 hook 前缀保持一致。
- **类型/接口**：用名词或名词短语，不加 `I` 前缀，不加 `Type` 后缀；Request/Response 成对出现时加对应后缀。

## 类型归属规则

- **`packages/types`**：跨 Desktop / Web / Mobile 三端共享的领域模型、API 契约类型放这里。新增前先确认确实需要跨端共享，不要把只在 Desktop 用的类型放进来。
- **`features/<domain>/shared.ts`**：只在某一功能域内使用的类型、枚举和常量放这里，不要上浮到 `packages/types`。
- **组件/Hook 文件内 inline**：只在单个文件内用一次的 props 类型或临时结构体可以 inline，超过两处引用就必须提升到 `shared.ts`。
- 禁止在页面文件或组件文件顶部定义大段类型，类型增多时先整理到 `shared.ts` 再继续开发。

## 跨端同步原则

**架构方向：以桌面端本地内核为核心。**

- Desktop 是旗舰端，Tauri/Rust/SQLite 本地内核承载所有核心业务能力；新功能默认先在 Desktop 实现和验证。
- Web 端定位为展示版 / 远程协作版，不追求与 Desktop 完全的功能对等；Web 端在 Desktop 功能稳定后选择性同步，不要在两端同时开发同一功能。
- Mobile 端定位为轻量陪伴端，只承载消费型场景（阅读、通知、快速查阅），不要求与 Desktop/Web 功能对齐。
- 跨端共享的业务逻辑、类型、工具函数放 `packages/` 下；端特有的 UI、Tauri 调用、Next.js 路由逻辑不得放入 `packages/`。
- Web 端同步时，如发现 Desktop 实现与 Web 存在结构差异，应优先对齐抽象层（hook / shared），而不是在 Web 端重新实现一套。
- Web/Mobile 连接同一 HTTP 后端服务（默认端口 8000），不得各自维护不同的默认连接地址。

## 状态边界

- **组件内 `useState`**：仅限 UI 临时状态（展开/收起、hover、输入草稿），不承载业务数据。
- **Hook 内状态**：异步请求结果、表单提交状态、派生数据统一放到对应 hook，页面/组件只消费 hook 返回值。
- **状态提升**：只在两个兄弟组件确实需要共享同一状态时提升到父级，不要提前提升；提升超过两层时，考虑抽成独立 hook 并通过 props 传入。
- 禁止在页面文件中用 `useState` + `useEffect` 组合直接管理异步数据；此类逻辑必须封装在 hook 内。

## Git 提交规范

- **格式**：`type: 中文描述`，描述简明扼要，一行说清改动意图。
- **类型**（type）：
  - `feat` — 新功能
  - `fix` — 缺陷修复
  - `refactor` — 不影响外部行为的重构
  - `chore` — 依赖更新、配置调整、脚本维护等杂项
  - `docs` — 仅文档变更
  - `ci` — CI/CD 流水线变更
  - `revert` — 回退某次提交
  - `release` — 版本发布（如 `release: v0.3.0`）
- **描述语言**：优先中文；技术专有名词（API、CI、macOS、WebKit 等）保持原文。
- **禁止**：不写「优化」「修改」「更新」等无实质内容的描述；不使用英文 subject（除非是纯 CI/配置文件的微小调整）；不省略 type 前缀。
- **多文件提交**：一个提交只做一件事；若同时涉及多个功能域，拆成多个提交分别提交。

## 验证要求

- 所有结构性重构至少通过相关包的 `type-check`。
- 涉及工作区行为变更时，同时通过仓库级 `pnpm type-check` 和 `pnpm lint`。
- 生成物、锁文件漂移和临时文件不得顺手带入提交。

## Agent 执行约束

- Claude、Codex 及其他代理在修改本仓库时，默认遵守本文件。
- 如果发现目标文件已经成为大文件，代理的第一步应是提出并执行拆分，而不是继续内联追加。
- 代理输出的实现应以“功能域目录 + 共享逻辑 + 组合页”作为默认组织方式。
