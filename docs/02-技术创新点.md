# 小妍 — 技术创新点

## 一、Supervisor 驱动的多 Agent 状态机编排

### 1.1 问题背景

传统单模型对话存在三个瓶颈：
- **能力瓶颈**：一个模型同时承担规划、检索、写作、分析，难以在每个子任务上都达到最优
- **上下文瓶颈**：长对话中历史消息不断累积，关键信息被淹没
- **可解释性瓶颈**：用户只看到最终结果，不知道答案从何而来

### 1.2 技术方案

小妍在 Rust 后端实现了**显式状态机驱动的 Agent 编排**：

```
用户输入 → Supervisor 解析意图 → 任务拆解 → 并行/串行调度 Agent → 结果整合 → 流式输出
```

核心设计：
- **显式状态图**：每个 Agent 节点有明确的输入/输出/状态转换，非黑盒调用
- **三种路由模式**：`rule`（纯规则）、`llm`（模型实时选路）、`hybrid`（规则初选 + 模型修正）
- **中间产物落库**：Agent 执行记录、计划步骤、Artifacts 全部持久化，支持复盘回放
- **SSE 流式传输**：前端实时接收执行状态更新，Mission Control 面板动态展示

### 1.3 创新价值

相比 LangChain / AutoGen 等框架的"链条式"编排，小妍的 Supervisor 架构更轻量、更可控、更适配科研场景的确定性需求。

### 1.4 七个专项 Agent 的角色分工

后端在 `agent_graph.rs` / `agent_nodes.rs` 中定义了 7 个专项 Agent，职责边界清晰：

| Agent | 职责 |
|---|---|
| **retrieval** | 图谱 + 语义检索，收集证据与溯源链（串行打底）|
| **planner** | 围绕主题生成分阶段、可执行的研究/学习路径 |
| **literature_scout** | 筛选最值得优先阅读的核心候选论文 |
| **survey** | 把检索结果组织成结构化文献综述 |
| **paper_analyst** | 解析单篇论文的研究问题/方法/实验/局限 |
| **reproduction** | 输出可执行、风险明确的复现链路 |
| **synthesis** | 汇总各步状态，组织为用户可直接使用的最终回答（串行收口）|

每个 Worker 的 system prompt 由 `assistant_prompts.rs` 的 `specialist_system(role, responsibility, extra_rules)` 模板动态生成，统一约束"不编造论文/数据、引用须可追溯"。

### 1.5 依赖图驱动的波次并行调度

小妍并非简单串行执行，而是按 `worker_dependencies()` 声明的依赖关系对选中 Agent 做**拓扑排序**（`compute_execution_waves`），同一波次内用 `tokio::spawn` **真并发**执行，Worker 之间不共享可变状态：

```
Wave 1: planner ‖ paper_analyst          （无上游依赖，并行）
Wave 2: literature_scout ‖ reproduction  （各自依赖上游产物）
Wave 3: survey                           （等待 planner + literature_scout）
```

整体节奏为 `Retrieval（串行）→ Worker 波次并行 → Synthesis（串行）`，墙钟时间取决于"最长依赖链"而非"Agent 总数"。

### 1.6 共享工作区与摘要级上下文传递

并发执行的 Agent 通过 `SharedWorkspace` 协作：每个 Agent 产出结构化 `AgentOutput`（`summary` / `full_content` / `metadata` / 耗时）。下游 Agent 通过 `build_worker_context()` **只读取上游的 summary 摘要**而非全文，在保证信息传递的同时显著压低 token 占用。

### 1.7 分层模型配置与成本闸门

- **三层模型**：Supervisor（路由，低温 `0.1`）/ Worker 基础 / 各专用 Agent，温度与模型可分别配置，解析优先级"专用 → 基础 → 全局默认"逐级回退。
- **成本控制**：`multi_agent_max_steps`（默认 6）限制单次最多执行的 Agent 数；`multi_agent_search_limit`（默认 8）控制 Retrieval 的检索深度。
- **全程可观测**：`Plan` / `RoutingDecision`（含路由策略、选中 Agent、决策理由、执行波次）/ `RunStarted` / `RunFinished` / `TextDelta` 事件经 Tauri 通道实时推送前端，并落库 `agent_runs` / `agent_artifacts` 支持复盘。

## 二、Graph RAG：知识图谱增强的检索生成

### 2.1 问题背景

传统 RAG（向量相似度检索）在处理科研问答时存在明显短板：
- 无法回答"某篇论文被哪些后续工作引用并改进"这类多跳问题
- 语义相似度高的论文不一定存在真实的学术关联
- 缺乏引用关系导致证据链不可追溯

### 2.2 技术方案

小妍实现了**双层检索架构**：

**第一层：向量检索**
- 论文全文分块 → Embedding 向量化 → 余弦相似度检索
- 支持论文库与知识笔记的联合检索

**第二层：图检索**
- 基于引用关系图引擎构建的论文引用网络
- 检索时融合**图邻域上下文**：不仅返回最相似的文本块，还带回其引用/被引用的论文片段
- 支持多跳关系推理

### 2.3 RAG 管线关键参数（`rag.rs`）

- **分块**：智能句界切分，`chunk_size=800` / `overlap=150`，UTF-8 字符边界安全；
- **Embedding**：配置优先级"独立向量服务 → 主 Provider 默认"逐级回退，支持 `text-embedding-3-small/large`、`BAAI/bge-m3`、Ollama 本地模型，批量生成（batch≈48）；
- **检索**：`cosine_similarity` 余弦相似度，`combined_search` 同时召回**论文块 + 知识笔记**并按分数取 `rag_top_k=5`；
- **存储**：向量以 JSON 存于 SQLite TEXT 列（`paper_chunks` / `knowledge_notes` / `memory_observations`），本地线性扫描——零外部向量库依赖、隐私自持，亦预留了未来切换 sqlite-vec/ANN 的优化空间。

### 2.4 数据规模

- **CCF 目录**：679 条会议/期刊内嵌索引
- **期刊分区库**：22804 条期刊（WoS / JCR / 中科院），支持动态过滤

## 三、视觉模型 + 规则引擎双路 PDF 图表解析

### 3.1 问题背景

PDF 中的图表是论文理解的核心，但现有工具普遍存在：
- 只能提取文字，无法提取图表
- 能提取位图但无法识别矢量图和表格
- AI 分析时从不引用图表编号

### 3.2 技术方案

小妍采用**双路并行提取 + 智能合并**策略：

**路 A：lopdf 规则提取**
- 使用 `lopdf` 库解析 PDF 内部对象树
- 提取内嵌的位图图像（XObject Image）
- 提取页面文本中的图表标题（正则匹配 "Figure N" / "Table N"）

**路 B：视觉模型扫描**
- 将 PDF 页面转为图片
- 调用配置的视觉模型（GPT-4o / Claude 等）逐页扫描
- 识别 lopdf 无法提取的矢量图、复杂表格
- 输出图表编号、类型、描述

**智能合并**：两路结果按图表编号去重合并，AI 精读时提示词要求积极引用图表编号。

## 四、本地自包含的 Tauri + Rust + SQLite 架构

### 4.1 架构选型对比

| 方案 | 代表 | 劣势 |
|---|---|---|
| 纯 Web + 云端后端 | Notion AI、ChatPDF | 论文需上传云端，隐私风险 |
| Electron + Node 后端 | Zotero | 包体积大，内存占用高 |
| **Tauri + Rust + SQLite** | **小妍** | **开发门槛高，但包小、性能高、本地优先** |

### 4.2 技术收益

- **包体小巧**：Tauri 应用包体远小于 Electron
- **内存高效**：Rust 后端性能优异，无 GC 停顿
- **完全离线**：SQLite 本地存储，无网络也能查阅已有论文和笔记
- **进程内运行**：所有 Rust 逻辑在 Tauri 主进程内运行，无独立后端服务启动成本
- **跨平台**：一套代码输出 macOS / Windows / Linux 三端安装包

### 4.3 数据安全设计

- API Key 存储：前端展示 `***` 掩码，Rust 后端持真实值
- 配置导出：AES-256-GCM + PBKDF2 加密
- 数据持久化：Application Support 目录，重装应用不丢失

### 4.4 LLM 传输层抽象

在 LLM 核心模块内部，小妍还实现了统一的传输层抽象：

- **多服务商统一接入**：OpenAI / Anthropic / DashScope / DeepSeek 等通过统一接口调用
- **SSE 流式抽象**：所有服务商的流式响应通过统一的流式管道处理，前端无需感知差异
- **视觉模型支持**：同一传输层同时承载文本和视觉模型调用

## 五、长期记忆注入机制

### 5.1 问题背景

通用 AI 助手缺乏对用户的长期认知：不知道用户的研究方向、不记用户过去的偏好、每次对话都是"第一次"。

### 5.2 技术方案

小妍实现了**三层记忆体系**：

| 层级 | 来源 | 用途 |
|---|---|---|
| 手动备忘 | 用户主动录入 | 重要研究方向、个人偏好 |
| 近期操作 | 系统自动记录 | 最近读过的论文、做过的分析 |
| 过程观察 | AI 自动提取 | 从对话中总结用户研究兴趣 |

**记忆注入**：小妍对话时自动检索相关记忆并注入上下文，让 AI "记得"用户的研究背景和偏好。记忆详情支持密码保护，保护研究轨迹隐私。

### 5.3 语义检索 + 混合打分（`memory_retrieval_service.rs`）

记忆检索不是单纯的向量相似度，而是**多因子混合打分**：

```
score = importance × 4              # 重要度基础分
      + 关键词匹配分                 # title/summary/narrative 分权重命中
      + cosine × 40 (SEMANTIC_WEIGHT) # 语义相似度（缺向量时自动跳过，退化为关键词）
      + 近期加权                     # 时效性提升
```

- **候选构造**：近 90 天记录（按时间取 150 条）∪ 历史已嵌入条目（最新 400 条），并集去重后重排；
- **鲁棒回退**：query 或候选缺少 embedding 时自动退化为纯关键词检索，不影响可用性。

### 5.4 存储健康治理与后台回填（`memory.rs`）

为避免记忆库无限膨胀、稀释检索质量，内置自动治理：

- **写入去重**：24h 内相同 `(source, event_type, summary)` 的观察跳过，防止反复打开同一论文刷屏；
- **过期清理**：`importance ≤ 1` 且超 90 天的低价值记忆自动删除；
- **总量上限**：超过 5000 条时按"重要度升序、时间升序"裁掉最旧最不重要的；
- **级联删除**：通过 `memory_events` 的 `ON DELETE CASCADE` 清理关联数据，避免悬挂；
- **后台回填**：embedding 分批异步补齐（batch=64），未配置向量模型时静默跳过，不阻塞主流程。

## 六、AI 模拟审稿的自然科学方法

### 6.1 技术流程

```
PDF 上传 → pdfjs-dist 本地全文提取 → 研究领域/方法关键词识别
→ 生成 2-4 位模拟审稿人（不同视角）→ 结构化审稿意见
→ 含分类标签 + 大修/小修/接收/拒稿结论
```

### 6.2 创新点

- **本地提取**：PDF 全文在本地提取，不上传原文到外部服务
- **多角色模拟**：不同审稿人从"方法论""实验设计""创新性"等不同维度评审
- **结构化输出**：评审结果直接入库，进入审稿归档跟踪流程
- **版本关联**：审稿意见与论文版本绑定，支持跨版本对比

## 七、联网搜索增强

### 7.1 问题背景

本地知识库和固定数据集的覆盖面有限，部分前沿问题需要实时检索最新研究成果。

### 7.2 技术方案

小妍通过内置的联网搜索能力，作为 RAG 和 Graph RAG 的补充检索通道。搜索结果经 LLM 整合后注入对话上下文，保证回答的时效性。

## 八、arXiv 三步级联智能检索

### 8.1 技术设计

arXiv 检索支持**三步级联筛选**，自动填充检索参数：

1. **研究领域选择**：计算机科学两级分类 + 其他领域平铺
2. **类型筛选**：全部 / 会议 / 期刊
3. **等级过滤**：CCF-A/B/C、中科院 1-4 区、Top 期刊、JCR Q1-Q3、SCIE/SSCI

动态等级通过查询本地 22804 条期刊库完成，选择后自动填充 arXiv 分类和期刊/会议范围。

### 8.2 LLM 重排

检索结果经 LLM 重排，优先推荐与用户研究方向最相关的论文。
