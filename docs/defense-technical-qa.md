# 小妍答辩 Q&A（学术评审版）

> 面向老师、同学与专家评委，侧重技术架构合理性、设计 trade-off、科研场景适配性与创新边界。回答避免代码细节，保留必要的技术概念与量化指标。

---

## 一、项目定位与研究动机

### Q1：为什么选“科研助手”这个方向？现有工具不是已经足够多了吗？

现有工具确实多，但大多是“单点能力”：ChatPDF 读论文、Zotero 管文献、ChatGPT 回答通用问题。研究生真正的痛点不是缺工具，而是工具之间的上下文不断丢失——读完的论文进不了对话，对话结论沉淀不到笔记，实验记录和投稿又是两个系统。

小妍的核心假设是：科研不是一连串独立任务，而是一个连续推进、需要反复收敛的过程。所以我们做的是一个围绕研究主流程设计的“研究操作系统”，而不是又一个聊天工具。

### Q2：你们的目标用户为什么是研究生和青年科研工作者？

因为这个群体有几个典型特征：

- 研究主题多变，需要频繁做选题、调研、精读、复现；
- 数据敏感，论文和实验数据通常不愿上传云端；
- 工具预算有限，对调用成本和本地可控性都很敏感；
- 工作流长但尚未固化，对“一体化”接受度高。

小妍的设计基本围绕这四点展开。

## 二、多 Agent 协同架构

### Q3：多 Agent 编排和单模型链式调用相比，核心优势是什么？

三个优势：

1. **能力适配**：规划、检索、精读、写作、复现指导对模型的能力侧重不同，专科 Agent 可以分别配置最合适的模型和提示策略；
2. **可解释性**：每个 Agent 的输入、输出、中间产物都结构化落库，用户能看到“结论是怎么来的”；
3. **可控性**：Supervisor（任务调度中枢）显式决定调用谁、不调谁，避免黑盒模型自由发挥导致的信息编造或无限循环。

### Q4：你们的 Supervisor（任务调度中枢）路由策略是怎么设计的？为什么默认用 hybrid？

三种策略：

- `rule`：基于关键词和意图硬规则，快、便宜、确定性高；
- `llm`：让模型实时判断，灵活但成本高、稳定性差；
- `hybrid`：先用规则初筛候选 Agent ，再用轻量模型做二次修正。

默认 hybrid 是因为科研任务的意图边界相对清晰，规则能覆盖大部分情况；但完全规则又容易漏掉复杂问法，所以用模型做“复核”，在成本、延迟和准确率之间取平衡。

### Q5：Agent 之间的并行调度是怎么保证一致性的？

通过显式依赖图。每个 Agent 声明自己依赖哪些上游产物，Supervisor（任务调度中枢）在运行前做拓扑排序，把无依赖的 Agent 分到同一波次并行执行，有依赖的必须等上游完成。Agent 之间不直接共享可变状态，而是通过一个共享工作区读取上游产物，避免竞争条件。

### Q6：上下文传递会不会带来很高的 token（模型计费与上下文长度单位）开销？

我们做了两层控制：

- 每个 Agent 产出时同时生成 summary 和 full_content，下游默认只读 summary；
- Retrieval 和 memory 检索都有 top-k 限制，避免把所有相关内容都塞进上下文。

实际单次多 Agent 任务的上下文增长是可控的，不会随着 Agent 数量线性爆炸。

### Q7：你们怎么防止 Agent 编造文献或结论？

两个主要机制：

1. **检索打底**：所有需要事实依据的回答都先经过 retrieval Agent（检索智能体）做图谱 + 语义检索，结论必须能追溯到本地论文库或知识笔记；
2. **提示词约束**：所有专科 Agent 的 system prompt（系统级提示词）都明确要求“不编造论文/数据，引用须可追溯”。

当然这不是 100% 保险，后续我们还计划加入引用一致性校验。

## 三、RAG 与 Graph RAG

### Q8：你们为什么认为科研场景需要 Graph RAG（图增强检索生成），而不是普通向量 RAG（检索增强生成）？

因为科研问题有大量“关系型”问题。例如：

- “这篇论文的方法被哪些后续工作改进了？”
- “这个领域最经典的 baseline 是什么？”
- “A 和 B 两篇工作之间有没有引用关系？”

这些问题靠语义相似度回答不了，必须依赖真实的引用关系。所以我们在向量检索之上构建了论文引用关系图，检索时把语义邻居和图邻居一起召回，再用模型整合，这就是 Graph RAG（图增强检索生成）。

### Q9：知识图谱是怎么构建的？需要人工标注吗？

目前主要基于论文元数据自动构建：从 PDF 或导入信息中提取标题、作者、引用关系，自动建节点和有向边。不需要人工标注引用关系，但节点质量依赖 PDF 解析的准确率。未来会考虑让模型辅助抽取更细粒度的“方法—改进”关系。

### Q10：向量存在 SQLite（本地轻量级数据库）会不会成为性能瓶颈？

当前采用线性扫描，对研究生个人常用的几百到几千篇论文规模完全够用。我们的设计原则是“先保证本地可用、零依赖、隐私自持”，性能优化作为后续可选项：未来可以无缝切换到 sqlite-vec 等本地 ANN（近似最近邻，向量加速检索）扩展，不需要改上层接口。

### Q11：你们的 RAG（检索增强生成）评估是怎么做的？

目前主要用两类指标：

- **检索侧**：召回率、top-k 命中率、引用来源可溯源比例；
- **生成侧**：回答的事实一致性、引用准确率、用户满意度反馈。

我们也内置了 679 条 CCF（中国计算机学会）会议/期刊索引和 2 万多条期刊分区数据，作为检索质量和学术相关性判断的参考。

## 四、本地架构与隐私

### Q12：为什么选择 Tauri（跨平台桌面应用框架） + Rust（系统级编程语言） + SQLite（本地轻量级数据库）这套技术栈？

这是一个在“工程可控性”和“用户体验”之间的 trade-off：

- Tauri（跨平台桌面应用框架）比 Electron（另一主流桌面框架）包体小、内存占用低；
- Rust（系统级编程语言）适合写长时运行的后端逻辑，无 GC（垃圾回收）停顿，也便于做并发 Agent 调度；
- SQLite（本地轻量级数据库）本地零配置、事务可靠、跨平台，而且不需要用户安装独立数据库。

代价是开发门槛比 Electron/Node 高，但换来的是本地优先、完全离线可用、隐私可控。

### Q13：本地架构下，多设备同步怎么做？

当前通过加密导出/导入实现配置和研究数据的迁移。文件采用 AES-256-GCM（对称加密算法） + PBKDF2（密钥派生函数） 加密，用户可以安全地把本地数据库和设置打包带走。未来如果做真正的多端同步，也会优先考虑端到端加密或本地局域网同步，而不是默认上云。

### Q14：API Key 真的安全吗？毕竟要调用外部模型。

API Key 必须离开本地去调用模型服务，这是不可避免的。我们做的是“最小暴露”：前端只显示掩码，真实密钥由 Rust（系统级编程语言）后端持有；配置文件导出时敏感字段也加密。风险点和所有使用第三方 API 的桌面应用一样，但用户的论文、笔记、实验数据不会上传。

## 五、PDF 图表解析与论文精读

### Q15：PDF 图表解析的双路策略是怎么想到的？

学术 PDF 里的图有两种形态：一种是内嵌位图，可以直接提取；另一种是矢量图或复杂表格，嵌在 PDF 对象树里很难规则抽取。所以我们用 lopdf（PDF 解析库）处理“能规则提取的部分”，用视觉模型处理“规则搞不定的部分”，两路合并后给 AI 精读使用。

这本质上是“工程成本低的路先走，成本高的路补漏”的分层思路。

### Q16：视觉模型识别图表的准确率怎么样？

视觉模型对“图中有几条曲线”“表格有几列”这类结构化识别效果不错，但对专业符号、公式、复杂流程图仍然可能出错。所以我们的设计是：视觉模型负责“发现图表位置和类型”，具体文字和编号仍优先依赖 PDF 文本层，两者互补。

### Q17：论文精读会输出哪些维度？

典型输出包括：研究问题、核心方法、实验设计、主要结论、局限性、复现提示，以及积极引用的图表编号。输出会保存为可复用的分析资产，后续对话和知识沉淀可以直接引用。

## 六、记忆体系

### Q18：长期记忆和简单聊天记录有什么区别？

聊天记录是按时间顺序的原始文本，容易冗长、主题混杂。长期记忆是经过结构化提炼的：手动备忘、近期操作、AI 观察三类来源，每条记忆有主题、摘要、重要度和时效性。对话时不是把整段历史塞进去，而是检索相关记忆注入上下文。

### Q19：记忆的“重要性”是怎么确定的？

目前混合三要素：用户是否手动标记、AI 对事件研究价值的判断、以及后续被引用/查看的频率。未来计划引入更明确的用户反馈信号，比如“这条记忆对我有用吗”。

### Q20：记忆库的自清理会不会误删重要内容？

清理策略比较保守：只删“低重要度 + 超过 90 天 + 总量超过阈值”的内容。高重要度或近期活跃的记忆不会被清理。另外，与研究主题、论文、笔记强关联的记忆通常重要度较高，优先级保留。

## 七、模型配置与成本

### Q21：不同 Agent 用不同模型，配置复杂度会不会太高？

我们提供三层回退机制：专用 Agent 模型 → Worker（工作节点）基础模型 → 全局默认模型。用户如果想精细控制可以逐个配置，如果嫌麻烦只配一个全局模型也能跑。这种设计兼顾了专业用户和普通用户。

### Q22：多 Agent 调用的成本大概是多少？

取决于任务复杂度和模型选择。我们以默认值估算：一次典型多 Agent 任务最多 6 个步骤，检索深度限制 8 条，调度用轻量模型，深度分析才用强模型。相比“让强模型从头到尾自由回答”，成本通常更低，而且答案质量更稳定。

### Q23：Supervisor（任务调度中枢）用低温设置有什么依据？

Supervisor（任务调度中枢）的任务是“稳定地做意图解析和路由”，不是生成创意内容。低温（0.1）可以降低路由决策的方差，让同样的输入更容易得到同样的调度结果，这对可解释性和调试都很重要。

## 八、三端与工程策略

### Q24：三端功能不对齐，会不会给用户造成困扰？

我们的设计原则就是“新能力默认落在桌面端，Web/Mobile 按需同步”。因为桌面端是本地内核，承载核心研究功能；Web 和移动只覆盖展示、阅读、通知等消费型场景。用户的核心研究流程仍然以桌面端为准，不会因为多端而 confused。

### Q25：你们的前端代码组织有什么特点？

核心原则是“低耦合、高内聚、拒绝大文件”：

- 页面只负责视图组合；
- 业务逻辑下沉到 `features/<domain>/`（按功能域组织的模块目录）；
- 副作用、Tauri（跨平台桌面应用框架）调用、异步请求统一封装成 hook（可复用的状态/副作用逻辑单元）；
- 同一功能域的常量、类型、纯函数收敛到 `shared.ts`（域内共享模块）。

这是为了保证项目长期可维护，避免功能越堆越乱。

## 九、具体模块与研究闭环

### Q26：工作台首页的智能推荐是怎么生成的？

工作台会综合用户当前的研究主题状态、论文导入数量、已分析数量、知识笔记、对话记录、投稿截止日期等信息，判断每个主题处于哪个阶段，然后推荐最合理的下一步行动。比如导入了很多论文但只分析了一篇，会优先推荐“继续精读”，而不是直接跳到聊天。

### Q27：实验记录和投稿管理为什么也要做进同一个产品？

因为科研的证据链是连续的：实验产生结果，结果支撑论文，论文投向会议，审稿意见又驱动下一轮实验。如果这些东西分散在不同工具里，就很难追踪“某个结论是从哪个实验来的、哪个版本论文投了哪里、审稿人怎么评价的”。小妍把它们串起来，形成可回溯的研究证据链。

### Q28：AI 模拟审稿的设计思路是什么？

审稿本质上是一个“多角色、多维度、结构化”的评审过程。我们让模型分别扮演方法论、实验设计、创新性等不同视角的审稿人，生成带分类标签和结论意见的评审报告。这样作者可以在投稿前预判问题，而不是等正式审稿意见回来才被动修改。

## 十、创新点与不足

### Q29：你们觉得小妍最大的技术创新是什么？

不是某一项单点技术，而是把几项技术按科研场景的需求重新组合：

- **多 Agent 显式编排**解决单模型能力边界和可解释性问题；
- **Graph RAG（图增强检索生成）**解决学术关系推理问题；
- **双路 PDF 图表解析**解决论文精读“读图”问题；
- **本地内核**解决隐私和离线可用问题。

组合起来形成一个科研场景下可用的闭环。

### Q30：目前的局限和未来工作是什么？

几个已知局限：

1. **PDF 解析准确率**：复杂版式、手写批注、扫描版 PDF 仍有提升空间；
2. **多 Agent 调度策略**：当前依赖手工设计的规则图，未来希望引入基于执行反馈的自适应路由；
3. **评估体系**：需要更大规模的科研用户测试来验证答案质量和效率提升；
4. **跨设备协作**：目前主要靠加密导出，真正的协作同步还在规划中。

## 十一、答辩建议

1. **从“科研痛点”讲起**：评委更容易理解“为什么做”，而不是“用了什么技术”。
2. **强调 trade-off**：比如本地 vs 云端、规则 vs 模型、SQLite vs 专用向量库，展示设计思考。
3. **用具体场景举例**：不要只讲架构，用“调研一个新方向”“精读一篇论文”“准备投稿”三个场景串起来讲。
4. **诚实谈局限**：评委更信任承认不足、有清晰未来计划的团队。
5. **准备 1–2 个 demo 锚点**：比如现场展示 Mission Control（Agent 执行监控面板）的 Agent 执行轨迹，或者 Graph RAG（图增强检索生成）的多跳问答效果。
