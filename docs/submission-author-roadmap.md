# 投稿人侧功能后续规划

## 产品边界

投稿模块只服务投稿人，不做审稿人端、编辑端或期刊工作流平台。

允许保留并增强的能力是“AI 模拟审稿 / 投稿前预审”，它的定位是帮助作者自检论文、降低盲投风险，而不是替代真实审稿流程，也不建设审稿人管理、审稿人评分、申诉仲裁、编辑派审等功能。

## 规划依据

后续规划基于科研投稿 APP 用户忠诚模型中的投稿人侧变量：

- 投稿焦虑下降
- 感知效率提升
- 服务期望确认
- 感知公平提升
- 投稿沉没成本转化
- 学术身份认同
- 学术归属感
- 学术社会资本

其中 MVP 优先抓住两个最强驱动变量：投稿焦虑下降、感知效率提升。

## 当前基础

桌面端投稿模块已经具备以下基础能力：

- 刊会追踪：会议 / 期刊收藏、筛选、补充自定义刊会
- 智能推荐雏形：基于方向、关键词和历史投稿推荐刊会
- 投稿看板：写作中、已提交、审稿中、录用、拒稿状态流转
- 提交清单：内容、格式、提交、合规类 checklist
- 版本管理：保存论文版本、上传 PDF、对比版本
- AI 模拟审稿：基于版本文本或 PDF 提取结果生成模拟审稿意见
- 审稿意见归档：记录真实审稿意见、处理状态、作者回复草稿
- Cover Letter / 摘要润色：辅助投稿材料准备

后续重点不是新增分散入口，而是把这些能力组织为投稿人的完整闭环：投稿前选择与自检，投稿中追踪与提醒，结果后转投与复盘。

## 不做事项

以下能力暂不纳入投稿模块：

- 审稿人关系管理
- 审稿人专业度 / 公平性公开评分
- 双向匿名审稿社区
- 期刊编辑工作台
- 审稿任务派发与审稿人库
- 独立仲裁或申诉委员会流程
- 面向会议 / 期刊主办方的投稿系统

如果未来需要“审稿质量记录”，也只作为作者自己的私有复盘字段，不做审稿人侧产品化。

## MVP 必做能力

### 1. 智能期刊匹配 2.0

目标：降低盲投焦虑，帮助作者找到身份匹配、风险匹配、周期匹配的目标刊会。

输入：

- 论文标题
- 摘要
- 关键词
- 研究方向
- 目标类型：会议 / 期刊 / 不限
- 目标档位：CCF、SCI 分区、中科院分区、用户自定义梯度
- 风险偏好：稳妥、均衡、冲刺
- 时间偏好：快速审稿、常规周期、不限

输出：

- 推荐刊会列表
- 匹配分
- 推荐理由
- 风险提示
- 可能拒稿原因
- 备选梯度：冲刺 / 主投 / 保底
- 下一步动作：加入追踪、创建投稿、生成预审报告

对应变量：

- 投稿焦虑下降
- 学术身份认同
- 服务期望确认

### 2. AI 投稿前诊断报告

目标：把现有 AI 模拟审稿升级为投稿前自检报告。

报告结构：

- 总体录用风险
- 创新性评估
- 方法可靠性评估
- 实验充分性评估
- 相关工作覆盖度
- 写作清晰度
- 目标刊会适配度
- 高风险问题
- 必须修改清单
- 建议修改清单
- 可暂缓修改项

交互要求：

- 支持从版本 PDF 自动提取正文
- 支持只输入摘要做轻量评估
- 支持选择目标刊会后做“面向该刊会”的诊断
- 诊断结果可转为 checklist
- 诊断结果可关联到版本记录

对应变量：

- 投稿焦虑下降
- 感知效率提升

### 3. 格式自动校对

目标：减少技术性拒稿与提交前低级错误。

优先做轻量版：

- 摘要字数检查
- 关键词数量检查
- 页数 / 字数检查
- 图表分辨率提醒
- 参考文献格式一致性检查
- 双盲匿名化检查
- AI 使用声明检查
- 利益冲突声明检查

后续增强：

- 按目标刊会模板生成动态清单
- 根据 PDF 元信息检测图像质量
- 根据参考文献文本识别格式混用

对应变量：

- 感知效率提升
- 服务期望确认

### 4. 投稿进度时间线

目标：把现有 Kanban 状态变成作者可理解、可预期的投稿过程。

核心节点：

- 写作准备
- 投稿材料检查
- 已提交
- 编辑初筛
- 外审中
- 意见返回
- 修改中
- 再提交
- 录用
- 拒稿 / 转投

每个节点记录：

- 开始时间
- 预计完成时间
- 实际完成时间
- 当前等待天数
- 下一步建议
- 是否需要提醒

对应变量：

- 服务期望确认
- 感知公平提升
- 投稿焦虑下降

### 5. 拒稿关怀与转投工作流

目标：拒稿后不让流程断掉，把沉没成本转化为下一轮投稿资产。

触发条件：

- 投稿状态变为 rejected
- 审稿意见中出现明显拒稿结论
- 用户手动点击“准备转投”

输出：

- 同档次备选刊会
- 降档稳妥刊会
- 需要补强的问题清单
- 转投 Cover Letter 草稿
- 可复用审稿意见
- 新版本修改计划

对应变量：

- 投稿焦虑下降
- 感知效率提升
- 投稿沉没成本转化

## 第二阶段能力

### 1. 审稿意见结构化解析

定位：作者侧处理真实审稿意见，不做审稿人产品。

能力：

- 把审稿意见拆成独立任务
- 自动分类：必须修改、建议修改、解释回应、语言润色、实验补充
- 每条意见关联处理状态
- 每条意见生成作者回复草案

对应变量：

- 感知效率提升
- 投稿沉没成本转化

### 2. Response Letter 助手

能力：

- 根据审稿意见生成逐条回复
- 支持语气选择：谦逊、正式、坚定、简洁
- 支持填入修改位置：页码、章节、图表编号
- 支持导出 Markdown / Word 草稿

对应变量：

- 感知效率提升
- 投稿焦虑下降

### 3. 版本与修改计划联动

能力：

- AI 诊断清单、审稿意见清单、格式清单统一进入“修改任务”
- 修改任务关联论文版本
- 保存“本轮修改说明”
- 生成版本差异摘要

对应变量：

- 投稿沉没成本转化
- 感知效率提升

## 第三阶段能力

### 1. 录用成就与发表档案

能力：

- 录用后生成成就卡片
- 生成发表记录
- 生成年度投稿报告
- 汇总投稿成功率、平均审稿周期、常投方向

对应变量：

- 学术身份认同
- 学术归属感

### 2. 学者投稿主页

能力：

- 投稿记录时间轴
- 代表作和投稿成果
- 常投方向
- ORCID 预留集成入口
- 可导出个人发表履历

对应变量：

- 学术身份认同
- 学术社会资本

### 3. 投稿经验资产库

能力：

- 保存用户自己的投稿经验
- 记录不同刊会反馈风格
- 记录审稿周期、修改轮次、结果
- 为后续期刊匹配提供个性化依据

对应变量：

- 服务期望确认
- 投稿沉没成本转化
- 学术社会资本

## 结构改造要求

当前投稿页面已经承担较多状态、远程调用、副作用和 JSX 组合。后续新增功能前应先拆分当前触达逻辑，避免继续把能力堆进页面。

建议拆分为：

- `useSubmissionDashboard.ts`：投稿列表、状态流转、进度时间线
- `useVenueRecommendations.ts`：刊会匹配输入、推荐、评分
- `useSubmissionPrecheck.ts`：AI 预审、PDF 提取、诊断报告
- `useSubmissionChecklist.ts`：格式校对、动态清单、完成度
- `useSubmissionVersions.ts`：版本历史、文件、差异与修改说明
- `useReviewResponse.ts`：审稿意见解析、回复草稿、处理状态
- `useRejectionRecovery.ts`：拒稿关怀、转投建议、转投材料生成

页面职责：

- 只负责路由级 tab、模块组合和少量 UI 状态
- 不直接承载 Tauri 调用、文件选择、AI 事件监听和状态机逻辑

## 推荐实施顺序

1. 拆分投稿页状态与副作用，先把当前能力整理到 hooks。
2. 升级智能期刊匹配，形成冲刺 / 主投 / 保底推荐。
3. 升级 AI 模拟审稿为投稿前诊断报告。
4. 增强格式自动校对，并把结果写入提交清单。
5. 将投稿看板升级为进度时间线。
6. 增加拒稿关怀与一键转投工作流。
7. 做审稿意见结构化解析和 Response Letter 助手。
8. 做录用成就、投稿履历和投稿经验资产库。

## MVP 验收标准

MVP 完成时，投稿人应能完成以下闭环：

1. 输入论文标题、摘要、关键词。
2. 获得目标刊会推荐，并理解推荐理由与风险。
3. 对选定目标刊会生成 AI 投稿前诊断报告。
4. 根据诊断报告和格式清单完成提交前检查。
5. 创建投稿条目并进入进度时间线。
6. 保存论文版本和投稿材料。
7. 被拒后获得转投建议、修改计划和转投 Cover Letter 草稿。

这个闭环完成后，投稿模块才算从“记录工具”升级为“投稿人陪伴系统”。
