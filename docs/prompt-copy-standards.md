# 提示与文案规范

本项目的用户提示、状态反馈、错误提示、空状态文案和模型 Prompt 统一遵循以下标准。

## 1. 适用范围

- 前端页面中的占位文案、字段说明、按钮文案、状态标签、空状态、错误提示、成功反馈。
- 后端会直接返回给前端的错误信息和说明文案。
- LLM 的 system prompt、task prompt、输出格式约束。

## 2. 用户文案标准

- 语气要求：专业、克制、清晰、可执行，不使用口语化、夸张化、情绪化表述。
- 占位文案：
  - 需要用户输入时，优先使用“请输入……”。
  - 提供示例时，统一使用“例如：……”。
- 错误提示：
  - 先说明结果未完成，再说明原因。
  - 如可恢复，给出明确下一步，例如“请检查网络连接后重试”。
- 空状态：
  - 无数据时使用“暂无……”。
  - 搜索无结果时使用“未找到……”。
  - 尽量补充下一步建议。
- 状态标签：
  - `pending` -> `待处理`
  - `running` -> `处理中`
  - `done` -> `已完成`
  - `failed` / `error` -> `失败`
- 标点规范：
  - 中文句子统一使用中文标点。
  - 避免连续省略号和感叹号。

## 3. 模型 Prompt 标准

- 结构统一为：角色 -> 任务目标 -> 判断边界 -> 输出格式 -> 禁止事项。
- 指令必须明确：
  - 输出语言
  - 输出格式
  - 证据边界
  - 不允许编造的信息
- 避免模糊表达，如“尽量”“最好”过多堆叠。
- 对 JSON 输出任务，必须显式要求：
  - 只返回合法 JSON
  - 不包含 Markdown、解释和代码块
  - 字段缺失时的回退策略

## 4. 文案示例

- 占位文案：
  - `请输入研究问题`
  - `例如：图神经网络在药物发现中的应用`
- 错误提示：
  - `请求未完成，请稍后重试。`
  - `请求未完成，请检查网络连接后重试。`
  - `未找到对应论文。`
- 空状态：
  - `暂无对话记录`
  - `未找到相关笔记，请尝试更换关键词。`

## 5. 落地要求

- 新增功能必须复用本规范，不再引入新的口语化或临时性提示。
- 修改现有功能时，优先同步检查相关的错误提示、空状态和 Prompt 是否一起符合规范。
