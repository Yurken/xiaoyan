# MinerU 接入未来规划

## 背景

当前桌面端已经具备文档导入、正文抽取、图片抽取、关键词生成、论文解读与复现指南等能力，但 PDF 解析链路仍存在几个长期问题：

- 不同提取器在不同论文上的表现差异明显，正文质量不稳定。
- 对复杂版式、扫描版、跨栏、公式、表格与图片混排文档的鲁棒性仍然不够。
- 当前链路以 PDF 为主，向 DOCX、PPTX、XLSX、图片等多格式扩展时缺少统一解析底座。
- 论文导入后的下游能力高度依赖解析质量，一旦正文顺序、段落结构或表格公式失真，关键词、解读、复现指南与 RAG 检索都会受影响。

`F:/MinerU` 提供了一个值得评估的候选方向。它更偏向“面向 LLM / RAG / Agent 的文档解析引擎”，支持：

- PDF、图片、DOCX、PPTX、XLSX 等多种输入格式
- Markdown、JSON 等结构化输出
- OCR、VLM、pipeline 等多后端模式
- 本地、CLI、REST API、Docker 等多种接入方式

本规划的目标不是立刻替换现有解析链路，而是为后续把 MinerU 逐步接入小妍系统提供一份可执行的路线图。

## 规划目标

### 目标

- 为桌面端建立“可切换、可回退、可比较”的文档解析能力层。
- 优先提升论文导入后的正文质量与结构质量，而不是只替换一个解析命令。
- 在不破坏现有桌面端本地内核架构的前提下，逐步引入 MinerU。
- 为未来多格式文档导入打基础，让解析能力不只服务 PDF。

### 非目标

- 不在第一阶段直接删除现有 `pdf_extract`、`lopdf` 与图片抽取链路。
- 不把 MinerU 接成“唯一解析路径”，避免一次性切换导致整体回退困难。
- 不在第一阶段引入重型远程服务依赖，桌面端仍以本地优先为原则。
- 不把本规划等同于立即上线 OCR 或 VLM 全量能力，模型与算力问题要单独评估。

## 当前系统中的接入位点

当前论文导入和解析的关键位置主要在桌面端 Tauri 后端：

- `apps/desktop/src-tauri/src/commands/papers.rs`
  - `papers_upload`
  - `papers_extract_pdf_text`
  - 导入后的预览提取、全文写库、分析、chunk、embedding、figure 提取
- `apps/desktop/src-tauri/src/commands/paper_text.rs`
  - 预览文本提取
  - 正文提取
  - `pdf_extract` 与 `lopdf` 的回退策略
- `apps/desktop/src-tauri/src/commands/paper_analysis_text.rs`
  - 关键词抽取、分析用文本清洗与章节切分
- `apps/desktop/src-tauri/src/commands/paper_artifacts.rs`
  - 论文目录与 `figures/` 产物目录管理

从职责边界看，MinerU 最适合先接到“文档解析能力层”，而不是直接侵入页面或上层产品流程。也就是说，未来应新增一个面向文档解析的 adapter 或 runtime 层，由它决定：

- 使用哪一个解析器
- 输出哪些原始产物
- 是否需要 OCR、版面分析、表格公式恢复
- 哪个结果作为下游分析输入

## 建议的接入原则

### 1. 先并行，后切换

MinerU 第一阶段应作为“并行候选解析器”接入，与现有 `pdf_extract` 和 `lopdf` 链路共存，而不是立刻替换主链路。

### 2. 先能力层，后产品层

先在 Rust 命令层和产物层打通 MinerU，再让关键词、解读、复现指南、RAG 等下游能力逐步消费 MinerU 的结果。

### 3. 先本地优先，后服务化

桌面端仍然坚持本地优先：

- 开发与实验阶段优先接本地 CLI 或 Python runtime
- 只有在本地封装成本过高时，才考虑 sidecar API 或独立服务进程

### 4. 先保真，后增量

接入成功的标准不是“多了一个工具”，而是：

- 正文质量更稳
- 结构更清楚
- 表格、公式、图片引用更可用
- 下游关键词、解读、复现指南质量实际提升

## 候选接入方案

### 方案 A：本地 CLI 适配

由桌面端在本地直接调用 `F:/MinerU` 或封装后的 MinerU 运行时，读取 Markdown 或 JSON 结果。

优点：

- 与当前桌面端本地解析命令层的接入方式最接近
- 便于本地调试与灰度
- 不必立刻维护常驻服务

风险：

- Python 依赖、模型依赖、平台差异带来的打包复杂度较高
- Windows、macOS、Linux 的运行时封装成本可能明显高于当前轻量 PDF 解析链路

适用阶段：

- Phase 0 调研验证
- Phase 1 小范围功能接入

### 方案 B：本地 sidecar API

将 MinerU 作为桌面端本地 sidecar 服务启动，Tauri 后端通过 HTTP 调用解析接口。

优点：

- 命令层与运行时解耦更彻底
- 更方便承载异步任务、状态查询与多后端切换
- 为后续批量解析、多格式扩展保留空间

风险：

- 进程管理、端口管理、生命周期管理更复杂
- 桌面端安装包与运行时的一致性要求更高

适用阶段：

- Phase 2 之后，如果 MinerU 能力被证明值得长期保留

### 方案 C：可选外部服务

允许高级用户在设置中接入外部 MinerU 服务地址，由桌面端把解析任务转发过去。

优点：

- 本地打包负担最小
- 适合高性能机器、实验室共享服务或远程 GPU 环境

风险：

- 偏离桌面端本地优先原则
- 会引入网络依赖、配置复杂度和数据流转风险

适用阶段：

- 明确有多人协作或重型 OCR、VLM 需求后再考虑

## 分阶段路线

### Phase 0：能力调研与基准验证

目标：确认 MinerU 是否值得接入，而不是基于印象推进实现。

建议产出：

- 准备一组代表性样本集
  - 学术 PDF
  - 双栏论文
  - 扫描版 PDF
  - 表格密集文档
  - 含大量公式的论文
  - DOCX、PPTX、XLSX 示例文档
- 对比以下结果
  - `pdf_extract`
  - `lopdf`
  - MinerU
- 统一评估维度
  - 标题与段落保真
  - 阅读顺序
  - 单词分隔质量
  - 表格与公式恢复质量
  - 图片与图注关联质量
  - 解析耗时
  - 失败率

阶段结论应回答两个问题：

- MinerU 是否在小妍常见文档上显著优于现有方案
- 值得先接哪一种输出：Markdown、JSON，还是二者都保留

### Phase 1：作为可选解析器接入论文导入

目标：在不破坏现有导入流程的前提下，把 MinerU 接成可开关的候选后端。

建议实现：

- 新增文档解析 adapter，例如：
  - `document_parse_runtime.rs`
  - `mineru_runtime.rs`
- 在 `papers_upload` 中增加解析器选择能力
  - 继续保留现有 `preview_text` 与 `full_text` 主流程
  - 新增 MinerU 产物，例如 `mineru.md`、`mineru.json`
- 将 MinerU 的输出接入统一的文本归一化与质量选择流程
- 在设置中增加实验开关
  - 关闭：完全不影响现有用户
  - 开启：导入时并行生成 MinerU 结果

阶段标准：

- 不影响现有论文导入成功率
- 新增产物可稳定落盘
- 对复杂论文样本，最终进入分析链路的正文质量明显提升

### Phase 2：把 MinerU 结果接入下游能力

目标：不仅存结果，还让系统真正从中受益。

建议接入范围：

- 关键词生成
- 小妍解读
- 复现指南
- chunk、embedding 输入文本
- figure、table、formula 的引用上下文

策略建议：

- 不是直接“全量信任 MinerU”
- 而是让下游能力消费统一的 `normalized source bundle`
- 由能力层统一选择最佳正文与结构化块

这一阶段更值得新增中间层概念：

- `source bundle`
  - 原始 PDF
  - `preview_text`
  - `full_text`
  - `mineru.md`
  - `mineru.json`
  - 图片与图表资产

阶段标准：

- 小妍解读空结果率下降
- 关键词脏词比例下降
- 复杂论文的复现指南可读性提升
- RAG 命中质量有可感知改善

### Phase 3：扩展到多格式文档导入

目标：MinerU 不只服务论文 PDF，而是成为桌面端统一文档解析能力的一部分。

可扩展范围：

- DOCX 导入
- PPTX 导入
- XLSX 导入
- 图片型材料导入
- 网页快照或外部资料解析

产品侧可能衍生的能力：

- 非 PDF 研究资料统一纳入知识资产体系
- 表格与幻灯片内容进入长期知识库
- 跨文档检索与引用体验更一致

### Phase 4：是否升级为长期默认方案

当且仅当以下条件成立时，再考虑把 MinerU 从“可选解析器”升级为默认链路：

- 三个平台运行时封装可控
- 常见文档样本质量稳定优于现有方案
- 性能开销可接受
- 失败回退机制完善
- 架构层没有把页面和上层流程重新耦合死

如果达不到这些条件，MinerU 也可以长期保持为：

- 复杂文档增强解析模式
- 实验功能
- 仅在特定格式上启用的专用后端

## 需要提前评估的关键风险

### 1. 运行时与打包复杂度

MinerU 的依赖体量与模型能力明显重于当前 `pdf_extract` 和 `lopdf` 链路。接入前必须评估：

- Windows、macOS 的可封装性
- 首次安装体积
- 更新策略
- 是否允许按需下载

### 2. 性能与超时

复杂解析能力通常更慢，尤其在 OCR 或 VLM 模式下。需要提前定义：

- 导入阶段的超时策略
- 前台与后台任务分工
- 失败后的自动回退路径

### 3. 结果一致性

MinerU 可能同时输出 Markdown、JSON、图片与中间结构。必须提前定义：

- 哪个文件是原始结果
- 哪个文件是归一化结果
- 哪个文件进入下游分析

### 4. 权限与隐私

如果未来支持外部 MinerU 服务，需要重新定义：

- 文档上传边界
- 本地与远端的责任划分
- 用户设置中的明确提示

## 建议的数据与产物组织方式

如果未来正式接入 MinerU，建议在论文目录下统一保存解析产物，而不是散落到单一路径中：

- `preview.txt`
- `full_text.txt`
- `mineru.md`
- `mineru.json`
- `figures/`
- `tables/`
- `artifacts.json`

其中 `artifacts.json` 可作为产物索引，记录：

- 解析器来源
- 版本信息
- 生成时间
- 文件路径
- 是否用于下游分析

这样后续不论是 UI 展示、回放调试还是解析回归，都更容易落地。

## 建议的下一步

短期内建议只做两件事：

1. 在本地基于 `F:/MinerU` 做一轮离线样本基准测试，拿到“值不值得接”的证据。
2. 如果结果成立，再进入 Phase 1，只把 MinerU 作为可选解析器并行接入论文导入链路。

这样推进的好处是：

- 不会一次性重写现有解析系统
- 不会把页面层和产品层重新耦合到具体解析器
- 能让 MinerU 的收益与成本都在真实样本上被验证

## 结论

MinerU 值得作为小妍未来文档解析底座的重点候选方案，但应以“并行接入、能力层隔离、逐步放量”的方式推进，而不是直接替换现有链路。

真正的验收标准不是“接上了 MinerU”，而是：

- 导入后的正文更干净
- 结构信息更可用
- 关键词、解读、复现指南与 RAG 结果更稳定
- 多格式文档能力有了统一演进路径

在这些目标被样本验证前，MinerU 应被视为未来规划中的重点方向，而不是立即默认的唯一实现。
