# 专利工具规划

## 定位

「专利工具」是小妍在科研全链路中向**科研成果转化与知识产权保护**方向的延伸，帮助用户在完成研究后高效地进行专利检索、专利性评估、交底书撰写与申请流程管理。

本工具定位为研究后期的辅助模块，与论文精读、实验记录、投稿管理形成互补：研究产出既可发论文，也可评估专利保护价值。

---

## 解决的核心痛点

| 痛点 | 说明 |
|---|---|
| 不知道自己的创新点是否可申请专利 | 缺乏专利性（新颖性、创造性、实用性）快速评估手段 |
| 专利检索门槛高 | 专业检索式（IPC/CPC/关键词组合）对科研人员不友好 |
| 交底书撰写困难 | 从技术方案到法律语言的转换成本高 |
| 专利申请流程零散 | 从交底书到审查意见答复涉及多个阶段，难以跟踪 |
| 论文与专利冲突 | 发表论文后丧失新颖性，需要提前规划披露时机 |

---

## 核心功能（待进一步确认范围）

### 方向 A：专利检索与 prior art 发现

- **自然语言检索**：用研究问题/技术方案描述直接检索相关专利
- **智能检索式生成**：LLM 根据用户输入生成 IPC/CPC 分类号 + 关键词组合
- **专利摘要解析**：提取权利要求、技术方案、附图说明、法律状态
- **相似度排序**：基于语义 + 权利要求重叠度排序最相关专利
- **prior art 地图**：以时间线或技术分支图展示相关专利分布

### 方向 B：专利性预评估

- **新颖性初筛**：检索是否已有完全相同或高度相似的公开方案
- **创造性提示**：提示最接近现有技术，并建议可争辩的创新点
- **披露风险提醒**：检测用户是否已在论文/博客/会议中提前公开
- **可专利性评分**：输出可参考的概率化评分与建议

### 方向 C：专利交底书与申请辅助

- **技术交底书生成**：从研究方案、实验记录、论文草稿中提取素材，生成结构化交底书
- **权利要求草稿**：生成独立权利要求与从属权利要求初稿
- **附图说明辅助**：基于实验图表生成附图说明文字
- **申请文件版本管理**：交底书、说明书、权利要求书的版本快照

### 方向 D：专利申请流程管理

- **申请时间线**：记录提交、受理、公开、实审、授权/驳回各节点
- **审查意见管理**：归档审查意见通知书，辅助生成答复草稿
- **费用与期限提醒**：年费、答复期限等关键 deadline
- **代理人与事务所对接**：导出标准化交底书包

---

## 推荐首期范围

考虑到小妍当前的核心定位是「科研全链路助手」，建议首期聚焦与研究工作流衔接最紧密的两个方向：

1. **专利检索与 prior art 发现**（与现有文献检索能力复用度高）
2. **专利性预评估**（直接回答"我的创新点能不能申请专利"）

交底书撰写与流程管理可作为二期，待一期验证用户需求后再扩展。

## 技术实现要点

### 数据源

| 数据 | 来源 |
|---|---|
| 中国专利 | 国家知识产权局公开数据库、知产宝、 incoPat（商用） |
| 全球专利 | Google Patents Public Datasets、Lens.org、EPO Open Patent Services |
| 分类号映射 | WIPO IPC / CPC 分类表 |
| 本地研究数据 | 论文库、实验记录、研究方向、知识图谱 |

### 检索与解析

- **检索式生成**：LLM 将自然语言转为布尔检索式 + 分类号
- **专利文本解析**：从公开 PDF/XML 中提取标题、摘要、权利要求、法律状态
- **语义匹配**：将专利摘要向量化，与用户研究方案做相似度排序
- **时间线聚合**：按申请日/公开日聚合同族专利

### 专利性评估流程

```
用户输入技术方案
    ↓
LLM 提取关键技术特征
    ↓
生成检索式并调用专利数据库
    ↓
获取相关专利列表
    ↓
对每个相关专利判断：
    - 是否公开了全部特征？→ 影响新颖性
    - 是否公开了部分特征？→ 组合后可能影响创造性
    ↓
输出评估报告：
    - 新颖性风险等级
    - 最接近现有技术
    - 建议争辩点
    - 披露风险提醒
```

### 数据模型建议

```sql
-- 专利检索任务
patent_search_tasks (
  id TEXT PRIMARY KEY,
  user_query TEXT,
  generated_query TEXT,    -- 生成的检索式
  patent_databases TEXT,   -- 查询的数据源
  status TEXT,
  result_count INTEGER,
  created_at INTEGER
);

-- 检索结果缓存
patent_results (
  id TEXT PRIMARY KEY,
  task_id TEXT,
  external_id TEXT,        -- 专利公开号
  title TEXT,
  abstract TEXT,
  applicants TEXT,
  inventors TEXT,
  ipc_codes TEXT,
  publication_date TEXT,
  legal_status TEXT,
  relevance_score REAL,
  raw_data JSON
);

-- 专利性评估报告
patentability_reports (
  id TEXT PRIMARY KEY,
  task_id TEXT,
  novelty_risk TEXT,       -- low | medium | high
  inventiveness_risk TEXT,
  closest_prior_art TEXT,  -- 相关专利 ID 列表
  key_features JSON,       -- 技术特征清单
  arguable_points TEXT,    -- 建议争辩点
  disclosure_risk TEXT,    -- 已公开风险提醒
  created_at INTEGER
);
```

## 与现有模块的关系

- **文献检索**：专利检索可与 arXiv / Semantic Scholar 检索共用检索编排与结果展示框架
- **论文精读**：精读论文时可选「评估专利性」入口
- **实验记录**：实验数据可作为专利交底书的素材来源
- **知识图谱**：专利技术特征可沉淀为知识节点
- **投稿管理**：在投稿前提醒用户评估是否需要先申请专利

## 待确认问题

在继续细化前，建议确认以下问题：

1. 目标用户是中国科研人员还是国际用户？（决定优先接入哪些数据源）
2. 首期重点是「检索」还是「评估」还是「撰写」？
3. 是否需要接入商业专利数据库（如 incoPat、PatSnap）？
4. 专利工具是否与律师事务所/代理机构有集成需求？
5. 是否需要在论文投稿前增加「专利优先申请」提醒？

- [ ] 对单个技术方案输出专利性预评估报告
- [ ] 检测用户是否已在本地论文/实验记录中提前公开相关技术
