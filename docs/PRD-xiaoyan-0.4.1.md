# PRD：小妍 0.4.1 研究闭环可用性增强版

## 1. Summary

小妍 0.4.1 的目标是把 0.4.0 建好的研究闭环底座变成更顺手的日常工作流。用户应该能从一个研究主题继续工作，清楚看到下一步，理解每个结论的证据来源，并把投稿诊断转成可执行的修改任务。

本版本不追求新增更多入口。重点是让已有规划、论文、知识、实验、投稿和小妍协同能力连得更清楚、用得更稳。

## 3. Background

### 3.1 Context

小妍已经覆盖科研全流程：

- 研究规划
- 文献综述
- 论文导入与精读
- 知识笔记与知识图谱
- 小妍多 Agent 协同台
- 实验记录
- 投稿管理
- 长期记忆与隐私保护

0.4.0 规划已经把方向定为“研究闭环稳定版 + 小妍内核收束”。它重点建设研究主题、论文解析质量、记忆 checkpoint、投稿诊断和 Agent Runtime。

0.4.1 要解决的是另一个问题：底座有了以后，用户能不能顺着它把研究持续推进下去。

### 3.2 Why Now

当前最大风险不是功能不够，而是用户路径不够清晰：

- 用户完成一次规划后，不一定知道如何继续调研。
- 用户读完论文后，不一定知道结论如何进入知识或投稿。
- 用户看到小妍的过程信息后，不一定知道哪些证据支撑了结果。
- 用户得到模拟审稿意见后，不一定能转成下一步修改任务。
- 新用户首次打开时，容易被多个模块和大量配置淹没。

如果 0.4.1 能把这些路径打通，小妍会从“能力很全的科研工具”进一步变成“可以长期陪用户推进研究的桌面工作台”。

### 3.3 What Became Possible

0.4.0 的底座让 0.4.1 可以更低成本地做体验闭环：

- 研究主题可以作为跨模块主对象。
- 记忆 checkpoint 可以提供“上次做到哪里”的上下文。
- Agent 事件流可以解释小妍做了什么。
- 投稿诊断可以沉淀为结构化报告。
- 解析质量摘要可以增强用户对论文分析的信任。

## 4. Objective

### 4.1 Objective

让用户能围绕一个研究主题持续推进工作，并在关键节点看清楚：

- 我现在在做什么？
- 上次做到哪里？
- 下一步该做什么？
- 这个结论来自哪些证据？
- 这个诊断如何变成修改任务？

### 4.2 Why It Matters

对用户：

- 少翻聊天记录，少跨页面找材料。
- 更容易恢复中断的研究任务。
- 更信任小妍给出的论文分析、综述和投稿建议。
- 更容易把 AI 结果变成真实行动。

对产品：

- 提升长期使用频率。
- 强化“研究操作系统”定位。
- 让桌面端旗舰路线更清楚。
- 减少继续堆功能导致的复杂度失控。

### 4.3 Alignment

0.4.1 继续遵守项目方向：

- Desktop 是旗舰端。
- 本地 SQLite 是核心资产底座。
- 页面只负责组合。
- 功能逻辑进入 `features/<domain>/`。
- 远程调用、Tauri、副作用、状态机进入 hook 或后端 service。
- 不新增大而全页面。

### 4.4 Key Results

| KR | 指标 | 目标 |
|---|---|---|
| KR1 | 继续研究任务完成率 | 80% 以上测试用户能从工作台恢复一个研究主题并找到下一步 |
| KR2 | 证据链理解率 | 80% 以上测试用户能说清一个 AI 结论来自哪些论文、图表或笔记 |
| KR3 | 投稿诊断转任务率 | 60% 以上诊断报告至少转出 1 条 checklist 或修改任务 |
| KR4 | 新手首个成功任务 | 70% 以上新用户能在首次使用中完成“导入论文 -> 查看精读结果” |
| KR5 | 工程质量 | 相关包 `type-check` 通过；跨工作区改动时 `pnpm type-check` 和 `pnpm lint` 通过 |

## 5. Market Segment(s)

### 5.1 Primary Segment：正在持续推进课题的研究生

他们的任务不是一次性问答，而是持续几周或几个月推进同一个方向。

主要工作：

- 找方向
- 读论文
- 写笔记
- 做实验
- 写论文
- 准备投稿

主要痛点：

- 资料散在不同工具里。
- 上次读到哪里容易忘。
- AI 给了结果，但证据和来源不够清楚。
- 研究任务被打断后很难恢复。

### 5.2 Secondary Segment：准备投稿的论文作者

他们已经有论文版本，需要更快发现风险并安排修改。

主要工作：

- 选择目标会议或期刊
- 检查论文风险
- 补实验
- 回复审稿意见
- 管理版本

主要痛点：

- 模拟审稿意见难转成具体修改计划。
- 实验结果、论文版本和投稿风险之间缺少连接。
- 拒稿转投时，历史判断和修改理由容易丢。

### 5.3 Secondary Segment：高强度使用 AI 的科研用户

他们会频繁用小妍协同台完成复杂任务。

主要工作：

- 让 AI 调研文献
- 让 AI 总结论文
- 让 AI 辅助写作
- 让 AI 做投稿和复现建议

主要痛点：

- AI 过程复杂，但用户只想知道步骤、证据和产物。
- 长对话后，重要决策难沉淀。
- 多 Agent 名词过多时，理解成本变高。

### 5.4 Constraints

- 0.4.1 继续优先 Desktop，不追求 Web / Mobile 全功能同步。
- 不引入账号体系、云协作或远程团队空间。
- 不把功能直接堆进现有大页面。
- 不做新的外部 Agent 身份。用户可见主体始终是小妍。

## 6. Value Proposition(s)

### 6.1 Continue Research Without Rebuilding Context

用户再次打开小妍时，不需要重新解释课题背景。小妍能展示最近决策、开放问题和下一步建议。

用户获得：

- 更快回到研究状态。
- 更少重复输入。
- 更少跨页面查找。

### 6.2 Trust AI Results Through Evidence

论文分析、综述和投稿诊断需要能追溯证据。用户可以看到结论关联的论文、图表、笔记、实验或版本。

用户获得：

- 更强信任感。
- 更容易检查 AI 是否理解正确。
- 更容易把结果用于真实写作和投稿。

### 6.3 Turn Diagnosis Into Action

投稿前诊断不只是一段建议。它要能转成 checklist、修改任务和补充实验。

用户获得：

- 更清楚的修改路线。
- 更少手动整理审稿意见。
- 更容易形成“问题 -> 修改 -> 证据 -> 新版本”的闭环。

### 6.4 Start With One Clear Path

新用户不需要先理解所有模块。第一次打开时，可以从“导入论文并完成精读”开始。

用户获得：

- 更低上手门槛。
- 更快看到核心价值。
- 更容易建立对小妍的第一印象。

## 7. Solution

### 7.1 UX / Prototypes

#### 7.1.1 工作台：继续研究

工作台新增“继续研究”区域，展示最近活跃主题。

每个主题卡片展示：

- 主题名称
- 最近活动时间
- 最近完成事项
- 开放问题
- 推荐下一步
- 快速入口：继续问小妍、查看论文、查看笔记、查看投稿

用户点击主题后进入主题工作区。

#### 7.1.2 主题工作区：研究主题 Command Center

主题工作区是 0.4.1 的主体验。

页面只做组合，数据由 `features/research-context/` hook 或后端 service 聚合。

首版布局建议：

- 顶部：主题名称、状态、最近活动、下一步建议
- 左侧：研究时间线
- 中间：当前重点材料，包括规划、论文、笔记、实验、投稿
- 右侧：最近决策、开放问题、小妍上下文入口

#### 7.1.3 证据链抽屉

在以下结果中提供统一证据链入口：

- 论文精读结论
- 综述段落
- 小妍回答
- 投稿诊断项

证据链展示：

- 结论
- 证据类型：论文、图表、笔记、实验、投稿版本、记忆 checkpoint
- 来源标题
- 摘要片段
- 跳转入口

#### 7.1.4 投稿诊断任务化

投稿诊断报告中，每条风险项可以转成：

- checklist 项
- 修改任务
- 补充实验
- 小妍追问

诊断项需要保留来源：

- 目标刊会
- 论文版本
- 诊断时间
- 引用的实验或论文证据

#### 7.1.5 新手主链路

首次使用时提供一个轻量路径：

1. 配置主模型。
2. 导入一篇 PDF。
3. 查看解析质量。
4. 生成论文精读。
5. 保存一个结论到知识笔记。
6. 让小妍基于这篇论文回答一个问题。

新手路径不隐藏其他模块，但避免一开始要求用户理解全部功能。

### 7.2 Key Features

#### Feature 1：继续研究入口

能力说明：

- 聚合最近活跃研究主题。
- 展示每个主题的最近活动、开放问题和下一步建议。
- 支持从工作台一键进入主题工作区。

验收标准：

- 用户能在工作台看到至少 3 个最近主题。
- 每个主题至少展示一个推荐下一步。
- 点击后能进入主题工作区，并保留主题上下文。

#### Feature 2：研究主题 Command Center

能力说明：

- 聚合主题下的规划、论文、笔记、实验、投稿和小妍会话。
- 展示时间线、最近决策和开放问题。
- 支持从主题跳转到相关模块。

验收标准：

- 主题页不直接写复杂异步逻辑。
- 数据聚合进入 `features/research-context/` 或后端 service。
- 用户能从主题页找到关联论文、笔记、实验和投稿。

#### Feature 3：统一证据链抽屉

能力说明：

- 为 AI 结果提供统一的证据展示组件。
- 支持论文、图表、笔记、实验、投稿版本和 checkpoint。
- 支持跳转到来源。

验收标准：

- 至少在论文分析和投稿诊断中接入。
- 每条证据包含类型、标题、摘要和跳转动作。
- 没有证据时明确提示“暂无可追溯证据”，不伪造来源。

#### Feature 4：投稿诊断转任务

能力说明：

- 诊断报告中的风险项可以转成 checklist 或修改任务。
- 修改任务可以关联论文版本和实验记录。
- 后续拒稿转投时可以复用诊断历史。

验收标准：

- 用户能从一条诊断项生成 checklist。
- 用户能从一条诊断项生成修改任务。
- 修改任务能显示来源诊断项和关联版本。

#### Feature 5：新手主链路

能力说明：

- 引导新用户完成第一次有价值的闭环。
- 优先跑通“模型配置 -> 论文导入 -> 精读 -> 保存结论 -> 小妍问答”。

验收标准：

- 用户可以跳过引导。
- 引导状态本地保存。
- 完成后不反复打扰用户。

#### Feature 6：小妍过程文案收束

能力说明：

- 协同台逐步减少用户必须理解的 Agent 术语。
- 以“步骤、工具、证据、产物”描述过程。
- 用户仍可展开查看更详细的运行信息。

验收标准：

- 小妍协同台默认视图能说明当前步骤。
- 每个完成步骤能展示产物或失败原因。
- 用户可见身份始终是小妍。

### 7.3 Technology

#### Frontend

建议新增或扩展：

- `features/research-context/`
  - `ResearchCommandCenter.tsx`
  - `ResearchTimelinePanel.tsx`
  - `ContinueResearchPanel.tsx`
  - `EvidenceDrawer.tsx`
  - `useResearchContext.ts`
  - `shared.ts`

- `features/submission/`
  - 扩展诊断报告 UI
  - 新增诊断转任务 hook
  - 复用现有 checklist、version、review 结构

- `features/copilot/`
  - 调整过程展示文案
  - 增加步骤、工具、证据、产物映射

页面约束：

- `Home.tsx` 只组合工作台区块。
- 新主题页只组合 Command Center 区块。
- 不把 Tauri invoke、节流、防抖、状态机放进页面 JSX。

#### Backend

建议新增或扩展：

- `services/research_context_service.rs`
  - 聚合主题详情
  - 返回最近活动
  - 生成下一步建议的基础数据

- `services/evidence_service.rs`
  - 统一证据类型
  - 查询证据来源
  - 提供跳转所需实体信息

- `services/submission_diagnosis_service.rs`
  - 保存诊断项
  - 诊断项转 checklist
  - 诊断项转修改任务

- `services/memory_checkpoint_service.rs`
  - 提供主题最近决策
  - 提供开放问题
  - 提供下一步摘要

Command 层只做参数校验、权限边界和 service 调度。

#### Data

如 0.4.0 已完成基础表，0.4.1 优先复用。

可能需要补充：

- `research_activity_events`
  - 记录主题下关键动作
  - 用于时间线和继续研究排序

- `evidence_links`
  - 记录结论和证据来源关系
  - 支持论文、图表、笔记、实验、投稿版本、checkpoint

- `submission_revision_tasks`
  - 如果 0.4.0 未完成，则在 0.4.1 落地
  - 保存诊断转出的修改任务

所有 schema 变更必须向后兼容，不破坏 0.3.x 和 0.4.0 数据。

### 7.4 Assumptions

| 假设 | 风险 | 验证方式 |
|---|---|---|
| 用户愿意以研究主题组织长期工作 | 部分用户只想按单篇论文工作 | 支持从论文反向进入主题，同时保留论文库主路径 |
| 下一步建议能提升继续使用率 | 建议可能太泛，用户忽略 | 先基于规则生成，再观察点击率和完成率 |
| 证据链能提升信任 | 证据太多会增加阅读负担 | 默认展示摘要，允许展开详情 |
| 诊断转任务能推动投稿修改 | 用户可能仍偏好手动管理 | 让转任务动作足够轻，不强制使用 |
| 新手主链路能降低门槛 | 高级用户可能觉得干扰 | 支持跳过，完成后不再出现 |

## 8. Release

### 8.1 Scope

0.4.1 是 0.4.0 之后的体验增强版本。

它不替代 0.4.0 的内核工作。它假设 0.4.0 已经至少完成：

- 研究主题基础关联
- 论文解析质量摘要基础能力
- 记忆 checkpoint 基础能力
- 投稿诊断报告基础能力
- 小妍 Agent 事件流基础能力

### 8.2 First Version

首版必须包含：

- 工作台“继续研究”入口。
- 研究主题 Command Center 的只读聚合版。
- 论文分析或投稿诊断中的证据链抽屉。
- 投稿诊断转 checklist 或修改任务。
- 新手主链路的最小版本。
- 小妍过程文案从 Agent 术语转向步骤和证据。

### 8.3 Future Versions

后续版本可以继续做：

- 更智能的下一步建议。
- 更完整的研究资产健康度。
- 更强的审稿意见结构化解析。
- Web / Mobile 的轻量主题视图。
- MinerU 或其他解析器的正式接入。
- 更完整的证据链编辑和人工校正。

### 8.4 Relative Timeline

建议按 3 个阶段推进：

| 阶段 | 时间 | 目标 |
|---|---|---|
| Phase 1 | 1 个迭代 | 主题聚合、继续研究、数据结构确认 |
| Phase 2 | 1 个迭代 | 证据链、诊断转任务、新手主链路 |
| Phase 3 | 1 个迭代 | 协同台文案收束、回归测试、发布收口 |

### 8.5 Release Checklist

- [ ] 相关桌面端包 `type-check` 通过。
- [ ] 跨工作区修改时 `pnpm type-check` 通过。
- [ ] 跨工作区修改时 `pnpm lint` 通过。
- [ ] 主题工作区、论文、投稿、小妍协同台完成手动回归。
- [ ] 新增 schema 经过旧数据兼容验证。
- [ ] `CHANGELOG.md` 增加 0.4.1 条目。
- [ ] 发布前完成 macOS 安装包验证。

### 8.6 Non-goals

- 不做云端协作。
- 不做账号体系。
- 不做审稿人端或编辑端。
- 不在 0.4.1 强制同步 Web / Mobile。
- 不把外部 Agent 身份暴露给用户。
- 不新增大页面，不继续向 `Papers.tsx`、`Copilot.tsx`、`Submission.tsx` 直接堆复杂逻辑。
