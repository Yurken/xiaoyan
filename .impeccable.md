# Impeccable Design Context — 小妍 Copilot

## Design Context

### Users
研究生（硕士/博士）在科研工作中密集使用。典型场景：写综述时需要高效梳理文献、跟进研究方向、沉淀笔记。用户认知负荷高，注意力宝贵，需要工具真正帮他们推进，而不是展示 AI 能力。

### Brand Personality
**沉浸 · 智能 · 可信赖**

- 让用户感到"有进展"——每次操作都有清晰反馈，研究积累可见
- 让用户感到"AI 真正帮到我了"——而不是"这是个 AI 工具"
- 克制、学术、有自己的气质——介于 Notion 的结构感与 Linear 的专业密度之间

### Aesthetic Direction
- **参考**：Notion（清晰的信息结构、克制排版）+ Linear（高密度专业感、精准的交互反馈）
- **反例**：AI 感过强的界面（蓝色渐变泛滥、Sparkles 图标堆砌、流光动效喧宾夺主）
- **主题**：仅 Light Mode，新拟态软阴影体系保留，但要更有层次感
- **色彩**：Apple Blue (#007AFF) 保留为主色调，但减少滥用；用结构和留白建立层次，而非颜色堆叠
- **圆角**：保留大圆角语言（24-28px），但需搭配更强的边界对比防止区域模糊

### Design Principles

1. **去 AI 装饰，留 AI 能力；AI 协作动效重设计而非删除**
   AI 功能通过结果和进度体现，静态装饰（Sparkles 图标堆砌、无意义蓝色渐变）要减少。但 AI 真正运行时的协作动效（信号流、进度光束、状态脉冲）是科技感的核心来源，必须保留并升级——方向是更精准、更克制、更有节奏感，而不是更多更花哨。动效只在 AI 实际执行任务时触发，idle 状态下收敛至最低强度。

2. **层次靠对比，不靠颜色**
   在新拟态体系中，区分不同区域/卡片/操作层级的核心手段是：阴影深度差、背景明度差、字重对比——不是靠叠加更多颜色。关键信息与次要信息对比度须达到 WCAG AA 级（4.5:1）。

3. **进展可感知**
   研究状态（待规划 → 生成中 → 已规划）、论文状态、笔记积累量——这些都是用户的"研究进度条"。设计要让这些状态在视觉上清晰且令人满足，而不是只用 Badge 文字区分。

4. **信息密度适当，布局克制**
   借鉴 Linear 的紧凑感，但保留 Notion 式的呼吸空间。列表项不要过度分散，卡片不要堆叠冗余信息。每个页面只有一个视觉焦点。

5. **有学术气质的个性**
   字体排版体现严谨：标题层级清晰、正文行距充足、数据/数字用 tabular 字体。整体气质接近"设计精良的学术工具软件"，而非"消费级 AI 产品"。

## Known Issues to Address
- 部分区域对比度不足，卡片与背景界限模糊（新拟态阴影过软导致）
- AI 味过浓：蓝色渐变、Sparkles 图标在非必要场景泛滥；AI 协作动效本身保留但需重新设计，使其更精准克制
- 侧边栏导航标签仅用 title 属性提示，缺乏视觉标签
- 工作台首页信息层级偏平，重点不突出

## Neumorphic Consistency Summary

- 当前主要不一致来自三类问题：
  1. 同类组件混用 token 样式、硬编码 `#E8ECF0/#C8CDD3/#FFFFFF` 和原生浏览器外观
  2. 卡片、按钮、图标按钮、下拉框分别存在多套阴影深度与边界强度
  3. 某些局部为了补视觉质感单独加重阴影，导致它们脱离整体页面语言

## Neumorphic Constraints

### Scope
- **柔和拟态** 只作用于 `data-style != "modern-minimal"` 的风格分支。
- **极简工作台** 保持平面化、边界化表达，不继承拟态额外阴影与高光。

### Component Rules
- 卡片统一分为三层语义：
  - `raised`：主容器 / 页面核心模块
  - `flat`：次级卡片 / 内部分组
  - `inset`：嵌入区 / 输入区 / 只读槽位
- 按钮统一分为三类：
  - `primary`：唯一主操作，可带明显体积感
  - `secondary`：次操作，使用较弱的 raised 阴影
  - `ghost`：文本或工具行为，不做独立拟态壳层
- 图标按钮必须与普通按钮共用同一套背景、边框、阴影 token，不能再单独硬编码一套拟态效果。
- 下拉框触发器、输入框、文本域必须使用同一组 `control` token；展开菜单使用统一的菜单背景与阴影 token。
- Badge/Chip 统一走 `chip` token，避免局部再手写一套小胶囊背景。

### Implementation Rules
- 禁止在业务组件中继续硬编码拟态色值和阴影：
  - `#E8ECF0`
  - `#EEF1F5`
  - `#F2F6FA`
  - `#C8CDD3`
  - `#FFFFFF`
- 优先复用 `packages/ui` 基础组件和全局 utility class，不在 feature 内重复实现 Card / Button / Dropdown / IconButton 外观。
- 新增拟态视觉时，先补 token，再补基础组件，最后才允许落到业务层。
- 所有对拟态的增强必须同步写出 `modern-minimal` 的平面回退，不允许只改一套风格。
