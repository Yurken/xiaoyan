# 小妍产品发现：当前产品改进点与新版本功能 Brainstorm

日期：2026-05-30

## 1. 产品机会判断

小妍当前已经不是单点 AI 工具，而是面向研究生与科研工作者的桌面端研究工作台。现有能力覆盖规划、综述、论文精读、知识沉淀、小妍协同、实验记录、投稿管理、工具集、长期记忆、应用锁与桌面伴侣。

下一阶段的主要机会不是继续横向增加入口，而是把已有能力从“功能集合”收束为“可持续推进科研任务的闭环”：

- 用户从一个研究主题出发，能连续推进规划、调研、阅读、笔记、实验、投稿。
- 每一步产出都能沉淀为可复用资产，并能被小妍在后续任务中正确引用。
- AI 过程不仅能生成结果，还要能解释证据来源、执行步骤、失败原因和下一步建议。
- 桌面端继续作为旗舰端，本地隐私、结构化资产和可观测 Agent 协同是核心差异化。

## 2. 当前功能的价值与可用性假设

| 功能域 | 当前价值假设 | 主要可用性风险 | 建议验证方式 |
|---|---|---|---|
| 研究规划 | 用户愿意从“研究主题”开始，让系统帮自己拆学习路线、关键文献和切入点。 | 规划结果如果不能自然进入综述、论文和知识，用户会把它当一次性文本。 | 找 5 位目标用户完成“输入方向 -> 生成规划 -> 继续调研”任务，观察是否能无提示进入下一步。 |
| 文献综述 | 用户需要快速形成结构化综述骨架，并希望引用本地论文库证据。 | 综述页若只输出长文本，用户难判断哪些论文值得导入或阅读。 | 对比“纯综述文本”与“综述 + 候选论文 + 争议点 + 后续动作”的完成率。 |
| 论文库与精读 | PDF 导入、全文提取、图表识别、AI 精读和复现建议是高频核心价值。 | 解析失败、图表缺失、分析不可追溯会直接伤害信任。 | 建立 10-20 篇代表 PDF 样本，记录正文长度、图表数量、失败原因、重解析成功率。 |
| 知识图谱与笔记 | 用户希望把阅读结论变成可复用知识网络，而不是散落笔记。 | 图谱如果不能回答“这个结论来自哪里”，会变成展示型功能。 | 让用户从一个结论反查证据论文、图表、实验记录，统计路径完成时间。 |
| 小妍协同台 | 多 Agent 可观测过程能提升 AI 可信度，是区别于普通聊天的重要卖点。 | 如果计划、Agent、工具、证据、产物概念过多，用户可能看不懂当前到底发生了什么。 | 观察用户能否在任务执行中回答“现在小妍在做哪一步、用了什么证据、失败时怎么办”。 |
| 实验记录 | 结构化记录和附件能把复现实验变成可追溯资产。 | 如果实验不能关联论文版本、投稿问题或知识结论，会退化为普通笔记。 | 让用户从模拟审稿问题创建补充实验，并回到投稿版本，验证闭环是否顺畅。 |
| 投稿管理 | DDL、版本、模拟审稿、审稿归档覆盖作者侧关键流程。 | 投稿页功能多，用户容易不知道“当前最该做什么”。 | 测试“选择目标刊会 -> 生成诊断 -> 转 checklist -> 创建修改任务”的任务路径。 |
| 工具集 | arXiv、分区、翻译、Markdown、PPT 等减少外部网站切换。 | 工具结果如果不能回写论文、知识、综述或投稿，长期价值有限。 | 统计工具结果被保存/引用/导入主流程的比例。 |
| 长期记忆 | 记忆能让小妍跨会话恢复研究上下文。 | 仅召回零散 observation 容易噪声大，也可能带来隐私顾虑。 | 比较“无记忆 / observation / checkpoint”三种注入方式下，续聊任务的准确率。 |
| 设置与模型分工 | 多模型分工让高级用户按成本和能力调度模型。 | 配置项过多会阻碍新用户首次跑通主链路。 | 首次启动可用性测试：用户能否在 5 分钟内完成模型配置并成功跑一次论文分析。 |

## 3. 失败假设：从 PM / 设计 / 工程视角看

### PM 视角

- 用户可能认可“科研全链路”叙事，但真实高频需求仍集中在论文精读、综述和投稿少数场景。
- 功能太全会让定位变得模糊，新用户可能不知道从哪里开始。
- 投稿、实验、知识图谱等重资产功能需要持续使用才能体现价值，冷启动阶段价值感不足。
- 多端叙事如果继续并行，会稀释桌面端旗舰路线。
- 新版本若继续加入口，可能扩大工程债，影响稳定性和发布节奏。

### 设计视角

- “规划、综述、论文、知识、实验、投稿”之间的上下游关系需要更强的路径提示，否则用户会在页面间迷路。
- 小妍协同台的信息密度高，计划、事件、Agent、附件、会话、产物需要更清晰的层级。
- 论文解析质量、记忆引用、Graph RAG 证据链都需要可视化解释，否则用户难以建立信任。
- 设置页应继续从参数面板转向任务配置流程。
- 桌面伴侣有品牌记忆点，但要避免抢夺主工作流注意力。

### 工程视角

- 当前仍存在大页面和大 command 文件：`Papers.tsx`、`Copilot.tsx`、`Submission.tsx`、`papers.rs`、`chat.rs` 等仍是后续改动高风险区。
- 研究主题、论文、知识、实验、投稿之间需要统一实体关联模型，否则闭环会靠前端拼接。
- Agent runtime、事件协议、工具层、context builder 如果不收束，新工具会继续以规则和特例方式生长。
- 论文解析需要 adapter 与质量元数据，否则接入 MinerU 或未来解析器会牵动旧链路。
- 记忆注入需要摘要、实体链接和隐私边界，否则会有 token 成本、回答漂移和安全风险。

## 4. 三视角 Brainstorm：新版本功能点

### Product Manager 视角

1. 研究主题详情页：把一个研究主题下的规划、论文、笔记、实验、投稿、小妍会话聚合成主对象。
2. 继续当前研究：工作台根据最近活动、未完成任务和开放问题推荐下一步。
3. 投稿前诊断报告：从目标刊会、论文版本、模拟审稿、实验记录生成结构化风险报告。
4. 研究资产健康度：展示某个主题的文献覆盖、实验证据、知识沉淀和投稿准备度。
5. 新手主链路引导：首次使用围绕“导入一篇论文并完成精读”而不是展示所有模块。

### Product Designer 视角

1. 研究时间线：把规划、导入、精读、笔记、实验、投稿动作按时间串起来。
2. 证据链抽屉：在论文分析、综述、投稿诊断中统一展示“结论 -> 证据 -> 来源”。
3. 小妍任务过程视图重构：把多 Agent 术语收束为“步骤、工具、证据、产物”。
4. 论文解析质量卡片：用明确状态展示正文提取、图表识别、章节识别、回退路径。
5. 设置向导：按“先跑通主模型 -> 再配置论文精读/综述/视觉模型”的任务流设计。

### Software Engineer 视角

1. AgentRuntime 抽象：统一请求、事件、工具调用、取消、结果归档。
2. ResearchContextService：统一聚合主题上下文、实体链接、最近活动和下一步建议。
3. PaperParserAdapter：隔离 `pdf_extract`、`lopdf`、视觉扫描、未来 MinerU。
4. Memory Checkpoint：新增会话摘要、实体链接、隐私分层召回和可控注入。
5. SubmissionDiagnosisService：把模拟审稿升级为可落库、可转任务、可关联实验的诊断能力。

## 5. Top 5 优先功能建议

### 1. 研究主题 Command Center

一句话：让“研究主题”成为串联规划、论文、知识、实验、投稿和小妍会话的主工作区。

为什么优先：

- 直接解决当前产品“功能多但闭环感不足”的核心问题。
- 能把已有模块的价值连起来，而不是新增孤立能力。
- 与 0.4.0 “研究闭环稳定版”主题高度一致。

关键假设：

- 用户愿意以研究主题作为长期组织单位，而不是只按单篇论文工作。
- 现有数据对象足以建立最小关联，不需要一次性重做全部 schema。
- 主题详情页提供下一步建议后，用户跨模块跳转成本会显著下降。

验证方式：

- 先做只读聚合 MVP：规划、论文、笔记、实验、投稿、最近会话、下一步建议。
- 找 3-5 位用户完成“从工作台继续一个研究主题”的任务，记录是否需要手动找上下文。

### 2. 论文解析与证据质量 2.0

一句话：给每篇论文建立解析质量摘要、重解析入口、解析器元数据和回退路径。

为什么优先：

- 论文精读是当前最核心、最高频、最能体现 AI 价值的入口。
- 解析失败会直接破坏用户对 AI 分析、图表引用和 RAG 的信任。
- 这是后续 MinerU、多解析器、图表增强的技术地基。

关键假设：

- 用户在看到质量摘要和失败原因后，会更愿意信任或修正分析结果。
- 保存解析运行元数据能显著降低排障成本。
- 现有解析链路可以被 adapter 包住，不需要大规模替换。

验证方式：

- 用 10-20 篇代表 PDF 建立解析基准。
- 记录每次解析的正文长度、图表数、耗时、失败原因、回退路径和用户重解析选择。

### 3. 记忆 Checkpoint 与“继续研究”

一句话：会话结束后沉淀目标、已完成事项、关键决策、开放问题和下一步，并与研究实体关联。

为什么优先：

- 小妍的差异化不只是回答，而是长期陪伴一个研究过程。
- checkpoint 比零散 observation 更适合恢复任务状态。
- 能自然支撑工作台的“继续当前研究”和主题页的“最近决策”。

关键假设：

- 用户需要跨天恢复研究任务，而不是每次重新描述上下文。
- 摘要式 checkpoint 足以改善续聊，不需要注入全文。
- 隐私分层可以降低对自动记忆的心理阻力。

验证方式：

- 设计同一任务隔天续聊测试，对比无 checkpoint 与有 checkpoint 的下一步准确率。
- 检查用户是否能从 checkpoint 找回上次决策，而不是翻聊天记录。

### 4. 投稿前诊断与修改任务闭环

一句话：把 AI 模拟审稿升级为诊断报告，并能一键转 checklist、修改任务和补充实验。

为什么优先：

- 投稿管理已经具备版本、审稿和实验基础，最缺的是“下一步怎么改”的决策层。
- 诊断报告能把论文、实验、投稿三个模块连接成强闭环。
- 对科研用户来说，这是接近真实高价值场景的付费/留存能力。

关键假设：

- 用户愿意在投稿前让 AI 做结构化风险诊断。
- 诊断项转成任务后，比纯文本审稿意见更能推动实际修改。
- 实验记录与版本关联能提升用户对证据链的重视。

验证方式：

- 用 3 篇论文样例生成诊断报告，让用户判断报告是否能转化为实际修改计划。
- 统计诊断项被转为 checklist 或实验任务的比例。

### 5. 小妍工具驱动 Agent Runtime

一句话：把小妍背后的规划、工具调用、事件流、上下文构造统一成 runtime，而用户仍只感知“小妍在执行任务”。

为什么优先：

- 当前多 Agent 可观测是亮点，但继续扩展会受到固定图和关键词规则限制。
- 统一事件流能让协同台更稳定地展示步骤、工具、证据、失败原因。
- 新增领域能力时，可以注册工具，而不是继续堆页面逻辑或 chat 规则。

关键假设：

- 用户关心过程可见，但不需要理解底层 Agent 名称和复杂路由。
- 现有固定 Agent 图能先被包装成默认 runtime，不影响旧行为。
- 工具事件比节点事件更适合解释“AI 做了什么”。

验证方式：

- 先抽象事件模型，不改用户可见行为。
- 在协同台 A/B 对比“Agent 节点视图”和“小妍步骤/工具/证据视图”的理解成本。

## 6. 建议的新版本范围

### P0：闭环骨架

- 研究主题详情聚合页。
- 工作台“继续当前研究”。
- 论文解析质量摘要和重解析入口。
- 会话 checkpoint 的写入、召回和实体链接。
- 投稿前诊断报告的保存与 checklist 转化。
- AgentRuntime / AgentEvent / Context Builder 的最小接口。

### P1：体验增强

- 研究时间线。
- 证据链抽屉。
- 研究资产健康度。
- 设置向导。
- 实验记录与诊断问题双向关联。

### P2：探索验证

- MinerU 实验性解析器入口。
- 更完整的审稿意见结构化解析。
- Web / Mobile 的轻量主题闭环展示。
- 更深入的 Agent 工具市场或技能管理。

## 7. 最小可验证实验

1. 主题闭环可用性测试：用户从工作台进入研究主题，找到上次读过的论文、最近小妍决策和下一步任务。
2. 解析质量信任测试：用户比较有/无解析质量摘要的论文分析结果，判断可信度和可修正性。
3. 续聊恢复测试：同一研究任务隔天继续，比较 checkpoint 注入前后的上下文恢复质量。
4. 投稿诊断转任务测试：用户选择论文版本和目标刊会，生成诊断并转为 3 个实际修改任务。
5. 协同台理解测试：用户观察一次小妍任务执行，回答“现在做到哪一步、用了什么证据、下一步是什么”。

## 8. 一句话版本主题建议

小妍 0.4.0 可以聚焦为：**研究闭环稳定版：从一个主题出发，把论文、知识、实验、投稿和小妍记忆串成可继续推进的研究过程。**
